代码设计与实现规范阅读路线

这份笔记接在《软件设计哲学》之后。目标不是记住更多术语，而是建立一套判断方式： 功能应该放在哪个模块，接口应该怎么设计，参数如何流动，测试如何快速定位问题。

核心问题 如何降低复杂度，而不是把代码拆得更碎。
工程目标 模块输入输出清楚，关键参数路径可追踪。
阅读方式 先读重构和实现规范，再读模式和架构。

第一梯队：直接提升代码设计

Refactoring: Improving the Design of Existing Code

最优先

重点是已经写出来的代码怎么一步步改好，同时保持行为不变。它适合现在这种复杂仓库： 先建立测试边界，再把坏结构慢慢拆开。

Code Complete, 2nd Edition

实现规范

更像实现手册。命名、函数、变量、条件、循环、调试、测试、代码布局都讲。 不需要通读，适合当工具书查。

The Pragmatic Programmer

工程习惯

它讲程序员日常判断：不要重复知识，避免偶然编程，快速反馈，自动化，测试，工具使用。 它不深，但很实用。

第二梯队：理解模式和模块边界

Head First Design Patterns, 2nd Edition

设计模式

如果想理解观察者模式、中介模式、工厂模式、策略模式、适配器模式，先读这本。 它比 GoF 原书容易读，适合建立直觉。

Patterns of Enterprise Application Architecture

架构模式

更工程化。讲分层、领域逻辑、数据访问、事务、服务层、数据映射。 对 cache、sampler、runner、env hook 这类模块关系有启发。

Architecture Patterns with Python

Python 架构

直接讲 Python 项目怎么拆：Repository、Service Layer、Unit of Work、Message Bus、 Dependency Injection。还有免费在线版。

第三梯队：维护已有复杂代码

Working Effectively with Legacy Code

复杂仓库

适合不是从零写漂亮代码，而是在已有系统里插入新机制的场景。 核心思想是先制造测试缝隙，再安全修改。

Clean Code

批判阅读

命名、函数职责、重复代码处理有价值。但它和《软件设计哲学》有冲突。 尤其是“函数越短越好”，不能机械接受。

更适合当前工作方式的判断是：模块要深，接口要小，内部可以复杂，但外部必须稳定。 不要为了短函数把一个完整参数处理链拆散到难以追踪。

第四梯队：Python 和实现规范

PEP 8

官方规范

Python 官方风格规范。重点看命名、import、注释、空行、行宽、public/internal interface。

Google Python Style Guide

团队规范

比 PEP 8 更工程化。适合团队代码，尤其是类型标注、异常、测试、导入、注释规范。

Fluent Python, 2nd Edition

语言进阶

用来把 Python 写得更地道。不建议太早读，它偏语言能力，不直接解决架构问题。

建议阅读顺序

  1. 先读 Refactoring，建立“改代码不破坏行为”的方法。
  2. 再用 Code Complete 补实现规范，按问题查，不必通读。
  3. 读 Head First Design Patterns，建立常见模式直觉。
  4. 读 Working Effectively with Legacy Code，学习在复杂仓库里安全插入新机制。
  5. 读 Architecture Patterns with Python，把模块边界和测试边界对应起来。
  6. 最后读 Patterns of Enterprise Application Architecture，理解更大的工程组织方式。
  7. 日常参考 PEP 8 和 Google Python Style Guide。
  8. 有余力再读 Fluent Python。

读这些书时要回答的问题

这个功能属于哪个模块？
这个模块的输入输出能不能单独测试？
关键参数在哪几个步骤被处理？
如果参数错了，我能不能逐层 probe？
接口是否稳定，内部是否可以替换？
这个抽象减少复杂度，还是只是增加名字？
测试是在检查行为，还是只检查能不能运行？
现在的拆分是否符合仓库已有设计模式？
