软件工程阅读路线

读代码设计书，不是为了背模式，而是为了少走弯路。

这份路线接在《软件设计哲学》之后。目标是形成一个稳定判断： 功能放在哪个模块，接口怎么收窄，参数怎么追踪，测试怎么定位问题。

第一目标 降低复杂度，而不是把代码切成更多碎片。
第二目标 模块输入输出清楚，关键参数路径能逐层 probe。
第三目标 先学重构和实现规范，再学模式和架构。
当前最重要的判断

模块要深。接口要小。内部可以复杂，但参数路径不能断。

不要机械学 Clean Code

“函数越短越好”不是铁律。对 FEMR 这种训练链路，过度拆分会让 `reference_window`、mask、reward、grad 这些参数更难追踪。

建议阅读顺序

按“先能安全改代码，再理解模式，再组织模块”的顺序读。

1

Refactoring

学习怎么在行为不变的前提下，把坏结构一步步改好。

2

Code Complete

补实现规范。命名、函数、条件、调试、测试都按问题查。

3

Design Patterns

建立策略、适配器、工厂、观察者、中介这些模式的直觉。

4

Legacy + Architecture

学习在已有复杂仓库里插入机制，并把模块边界和测试边界对齐。

第一梯队：立刻改善代码

这三本最直接。它们解决的是每天写代码都会遇到的问题。

Refactoring: Improving the Design of Existing Code

最优先

重点不是“写漂亮代码”，而是已经写坏的代码怎么安全变好。 它适合 FEMR 这种已有仓库：先建立测试边界，再逐步移动责任。

读它时看：如何拆出 seam，如何保持行为不变。

Code Complete, 2nd Edition

实现规范

更像实现手册。命名、变量、函数、条件、循环、调试、测试、布局都讲。 不需要通读，遇到具体问题时查。

读它时看：怎么让代码更容易检查。

The Pragmatic Programmer

工程习惯

讲日常判断：不要重复知识，不要偶然编程，快速反馈，自动化，工具使用。 它不深，但能纠正很多低效习惯。

读它时看：如何减少心理省力、实际费力。

第二梯队：理解模块边界

这些书帮助判断“这个责任属于谁”。不要为了套模式而套模式。

Head First Design Patterns, 2nd Edition

模式直觉

适合先读。它把策略、适配器、工厂、观察者、中介讲得比较直观。 读完后再看 FEMR 的 runner、env hook、sampler 会更容易。

读它时看：模式解决的是哪种变化。

Architecture Patterns with Python

Python 架构

直接讲 Python 项目怎么拆：Repository、Service Layer、Unit of Work、 Message Bus、Dependency Injection。在线版可读。

读它时看：业务对象和外部框架怎么隔离。

Patterns of Enterprise Application Architecture

架构模式

更工程化。讲分层、服务层、数据映射、事务脚本。对 cache、sampler、 runner、env hook 的责任划分有启发。

读它时看：模块如何协作而不互相污染。

第三梯队：维护复杂仓库

FEMR 更像“在已有系统中插入机制”，不是从零写一个干净项目。

Working Effectively with Legacy Code

复杂仓库

核心是先制造测试缝隙，再安全修改。它非常适合训练代码、runner、 checkpoint、env hook 这种互相缠住的路径。

读它时看：先保护，再重构。

Clean Code

批判阅读

命名、函数职责、重复代码处理有价值。但它和《软件设计哲学》有冲突。 不要机械接受“短函数”。

读它时看：哪些建议会破坏参数追踪。

Fluent Python, 2nd Edition

语言进阶

用来把 Python 写得更地道。它偏语言能力，不直接解决架构问题。 建议在模块设计稳定后再读。

读它时看：如何写更清楚的 Python。

规范怎么用

规范不是目的。规范是为了让别人和未来的自己更快定位问题。

材料 什么时候看 在 FEMR 中对应什么
PEP 8 命名、import、空行、public/internal interface 不确定时。 让模块边界更清楚，避免 helper 和 public API 混在一起。
Google Python Style Guide 团队代码、类型标注、异常、测试命名不确定时。 让 contract test、fake env、probe 输出有统一形态。
runtime-probing-debug 形状、mask、reward、grad、checkpoint 出问题时。 沿一个参数在模块内部逐层 print/assert，而不是只测能否启动。

读书时要一直问的问题

这些问题比记住术语更重要。

这个功能属于哪个模块？

数据对象、采样、reset、reward、storage、loss、runner 调度要分清。

输入输出能单独测试吗？

不能单独测试，就很难知道参数从哪一步开始错。

关键参数在哪些位置被处理？

例如 `reference_window`、`valid_mask`、`reward`、`requires_grad`。

接口是否稳定？

外部接口要小，内部实现可以复杂，但不能让调用方知道太多细节。

这个抽象减少复杂度了吗？

如果只是增加名字，没有缩短参数追踪范围，它就不是好抽象。

测试是在测行为吗？

只证明能运行不够。要证明语义正确，最好有 probe 和 assert。
